Comparison: Emergency Mesh vs. Bitchat and other mesh messaging apps

Focused on networking approach, dependency model, and developer ergonomics.

transport topology server dependency E2EE

What Emergency Mesh implements today (in this repo)

Web (emergency-mesh)

  • Local “mesh” transport: browser BroadcastChannel (same origin), useful for multi-tab simulation.
  • Discovery: ANNOUNCE/HEARTBEAT + peer timeout.
  • Features: chat + SOS broadcast/ack.

Mobile (mesh-mobile)

  • Transport: WebSocket client to /api/ws/mesh (relay/broadcast).
  • Offline content: posts cached in AsyncStorage with retention cutoffs + expiry.
  • Sync model: optional “internet sync” pull/push path for posts.

Note: this repo does not currently implement BLE/Wi‑Fi Direct multi-hop routing or end-to-end encryption. The “mesh” layer here is a relay + local broadcast simulation.

Implications for this project

Why Emergency Mesh is different

  • Ops-first: a web dashboard + mobile client + API server gives a place to coordinate incidents.
  • Extensible: OpenAPI/Zod types make it straightforward to add new message types, map layers, or retention rules.
  • Prototype-friendly: the web app simulates “nearby peers” across tabs for fast iteration.

If you want Bitchat-like properties

  • Mesh transport: add BLE / Wi‑Fi Direct + multi-hop routing on-device.
  • Security: add E2EE with modern primitives and authenticated peer identity.
  • No relay: redesign WebSocket broadcast into optional infrastructure, not a dependency.
